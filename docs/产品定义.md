# CampLus 产品定义

> 最后更新：2026-06-19
> 
> 本文档定义 CampLus **是什么、解决什么问题、由哪些核心系统组成**。
> 所有后续的路由、组件、数据模型、Phase 规划均从本文档派生。

---

## 一、CampLus 是什么

**一句话**：CampLus 是校园学术生活的操作系统——把学生与"课程、知识、选课、信息获取"相关的所有需求，从碎片化的 QQ/微信/Piazza/教务系统中抽出来，整合到一个结构化、可靠、不吵闹的平台里。

**核心哲学**：

1. **学生自治是战略选择**——平台在没有任何教师参与的情况下完整运转。课程信息由学生自主上传、集体编辑、集体审核、集体维护。教师是可选接入的"权威确认者"，不是系统运转的前提。
2. **事实优先于情绪**——这是一个围绕"事实、时间与责任"构建的系统，不是表达情绪的社交平台。
3. **信息积累而非信息流**——内容被结构化存储、可引用、可追溯、可继承。不是"阅后即焚"的聊天记录。

---

## 二、解决的痛点

| # | 痛点 | 现状 | CampLus 的解法 |
|---|---|---|---|
| 1 | **课程信息碎片化** | 一门课的信息分散在 QQ 群公告、群聊天、Piazza、教务系统、教师个人主页 | 每门课一个结构化仪表盘，所有信息集中 |
| 2 | **行政通知被 QQ 淹没** | 辅导员/教务处/学院的通知和闲聊混在一起，重要信息被刷掉 | 待设计（组织模式尚未想清楚） |
| 3 | **选课靠玄学** | 不知道老师的授课质量、给分分布、课程工作量、时间是否冲突 | 课程搜索引擎 + 档案页（成绩分布/评价聚合）+ Schedule Builder |
| 4 | **评课数据不流通** | 现有教学评估是行政导向的，数据不公开，对后来学生无参考价值 | 结构化评价系统，数据公开，支撑选课决策 |
| 5 | **社团信息孤岛化** | 每个社团一个 QQ 群+公众号，需要"打入内部"才能了解 | 组织公开主页 + 活动管理 + 公告发布 |
| 6 | **学生笔记锁在本地** | 好的笔记、总结、复习资料只存在于个人电脑或群文件里 | 知识广场：Notion 式编辑 + GitHub 式协作，可被课程引用和发现 |

## 三、五大核心系统

### 系统 A：课程仪表盘（Course Dashboard）

**解决痛点 #1**

每门课程拥有一个结构化主页。核心组成：

- **课程基本信息**：代码、名称、院系、学分、类别
- **学期列表**：历史学期进入只读归档，当前学期活跃
- **Timeline**：按教学周组织的节点（公告、作业、考试、资料、里程碑），是课程主页的信息核心
- **讨论**：围绕 Timeline 节点的上下文讨论（Q&A），不提供脱离节点的自由发帖
- **版本历史**：所有编辑记录 + Diff + 回滚

**编辑模式**：
- 学生自主上传和维护，标注信息来源（"课堂宣布于 3/15" / "学生整理"）
- 集体编辑 → 维护者审核合并（类似 GitHub PR 流程）
- 教师可标记"官方确认"，但不是必要环节
- 所有编辑永久可追溯

---

### 系统 B：个人首页（Personal Homepage / Dashboard）

**解决痛点 #6（个人学术全景缺失）**

登录后的默认页面。**仅展示与个人课程强相关的内容**：

- **我的课程卡片**：本学期在修课程，每门课显示下一个截止时间、本周要点
- **待跟进清单**：跨课程的作业和截止时间汇总
- **课程通知聚合**：被订阅课程的更新（新节点、修改、截止提醒）
- **学业进度总览**：已完成学分 / 总需学分 / 各模块完成情况

**明确不包含**：行政通知、全站信息流、推荐内容。

---

### 系统 C：知识广场（Knowledge Square）

**解决痛点 #6（笔记锁在本地）**

独立的、类似 Notion + GitHub 的知识协作平台。

**核心特征**：
- **BlockNote 驱动**：块级编辑器，自由组织内容。Markdown + LaTeX 原生支持
- **GitHub 式协作**：不是 Google Docs 的实时共写。每个知识库有维护者，修改通过"提交 → 审查 → 合并"流程。有变更历史和回滚能力
- **与课程的关联是双向链接，不是隶属**：
  - 课程 Timeline 的 material 节点可以**引用**知识广场的笔记（"推荐阅读"）
  - 知识广场的笔记可以**引用**课程 Timeline 的节点（"来源"）
  - 课程主页可以展示"关联的社区笔记"
  - 个人主页可以 pin 自己的知识库
- **内容性质**：学生个人或团队的笔记、总结、复习资料、知识整理。不承担"事实准确性"责任——那是课程 Timeline 的事
- **可以独立运作**：即使脱离其他 CampLus 系统，知识广场本身就是一个有用的"校园版 Notion/GitBook"

**与课程 Timeline 的边界**：

| | 课程 Timeline | 知识广场 |
|---|---|---|
| 内容来源 | 课程维护者集体整理 | 任意学生创建 |
| 内容形态 | 结构化节点（有 schema） | 自由形态文档/知识库 |
| 编辑方式 | PR → 维护者合并 | PR → 知识库维护者合并 |
| 权威性 | 追求事实准确，标注来源 | 署名但不声称权威 |
| 责任 | 维护者承担审核责任 | 作者署名，自行负责 |

### 系统 D：选课 × 评课（Course Discovery & Evaluation）

**解决痛点 #3 + #4**

选课和评课是**一个闭环的两个方向**：评价产生数据 → 数据支撑选课 → 选课后的学期产生新评价。

**包含四个子模块**：

1. **课程搜索引擎**
   - 按课程名称/代码/院系/学分/类别/时间筛选
   - 模糊搜索 + 通配符支持
   - 搜索结果直达课程档案页

2. **课程档案页**（参考 Atlas）
   - 基本信息 + 历年开课记录
   - 历年成绩分布（各分数段比例）
   - 评分聚合（教学质量、课程内容、难度、工作量）
   - 教授评价聚合（清晰度、尊重度、备课充分度等维度）
   - 数据可视化呈现

3. **Schedule Builder**（参考 Atlas）
   - 将候选课程加入规划表
   - 可视化日历网格视图，生成所有可能的课表排列
   - 自动检测时间冲突
   - 锁定心仪时间段，屏蔽忙碌时段
   - 生成课表，导出/分享（暂不接入教务系统选课）

4. **教学评价系统**

   评价分为两层：**过程跟踪 + 最终评价**。

   **过程跟踪（学期中）**：
   - 修读期间可以随时记录对课程的印象、感受、笔记
   - 形式：私有的阶段性 check-in（打分草稿 + 文字笔记）
   - 可以随学期进展持续更新
   - 仅供自己回顾，不公开、不参与聚合

   **最终评价（学期结束后）**：
   - 触发条件：学期结束 + 用户标记为"已完整修读"
   - 评价结构：定量打分（匿名展示，系统可追溯）+ 定性论述（实名）
   - 约束：提交后不可修改、教师可回应不可删除
   - 数据用途：聚合为课程档案页的数据来源

   **两层关系**：过程跟踪是私人的、辅助性的——帮助学生积累素材，最终评价时可以参考自己之前的记录。最终评价才是正式数据。

---

### 系统 E：组织与社团（Organizations & Clubs）

**解决痛点 #5**

与课程系统同级的一级模块。

- **组织公开主页**：简介、成员、负责人、联系方式
- **活动管理**：每场活动是一个独立条目——标题、时间、地点、描述、状态（预告/进行中/已结束/取消）、可选报名/签到。活动之间没有阶段递进关系，**不复用课程 Timeline 结构**
- **组织公告**：社团的公开通知（招新、公告等），区别于活动（活动有时效性和地点，公告是纯信息发布）
- **成员可见列表**

**为什么不复用 Timeline**：课程 Timeline 的核心是"周→周→周"的递进结构，节点之间有先后依赖和累积关系。社团活动更接近"行政通知"——每场活动是独立的、离散的，没有"阶段1→阶段2→里程碑3"的叙事。

---

### 待设计：行政通知系统

**解决痛点 #2**

核心矛盾已识别：量大且重要性不一，学生不能选择不接收，但按邮件组织会太乱，按公告板又不够。设计方案待定，但已确认**不在首页展示**，为独立模块。

## 四、系统之间的关系

```
个人首页 ──→ 聚合来自"课程仪表盘"的更新
   │
   ├──→ 点击课程卡片进入 → 课程仪表盘
   │                          ├─ Timeline 节点 ──→ 链接到 → 知识广场笔记
   │                          ├─ Material 节点 ──→ 引用自 → 知识广场笔记
   │                          └─ 讨论（围绕节点）
   │
   ├──→ 知识广场（独立入口）
   │        └─ 笔记 ──→ 引用 → 课程 Timeline 节点
   │
   ├──→ 选课×评课
   │        ├─ 课程搜索 → 课程档案页（成绩+评价数据）
   │        ├─ Schedule Builder（可视化排课）
   │        └─ 教学评价 → 数据流入课程档案页
   │
   └──→ 组织与社团
            └─ 活动与公告（独立数据模型）
```

**关键交互**：
- 课程 ↔ 知识广场是**双向链接**关系，不是隶属。通过超链接/引用关联，不通过数据所有权
- 选课×评课的数据来源于教务系统 + 评价系统，产出到课程档案页
- Timeline 是课程系统的核心数据结构，但不被组织/社团模块复用——不同实体有不同的时间组织方式

## 五、核心设计原则

### 5.1 学生自治

> **平台在没有任何教师参与的情况下完整运转。**

- 课程信息由学生上传、编辑、审核、维护
- 教师是可选接入的"权威确认者"，可以标注官方信息、修正事实错误，但：
  - 不能删除学生编辑的内容
  - 不能删除评价
  - 不能禁止讨论
- 这样做的原因不只是理想——是**战略选择**：一旦系统运转依赖学校行政配合，就会被卡住

### 5.2 责任可追溯

- 所有编辑记录编辑者、时间、修改说明
- 历史版本不可删除
- 讨论实名
- 任何获得更大权限的角色，必须同时承担更明确、可追溯的责任

### 5.3 信息结构化而非信息流

- 没有全站信息流
- 没有热度排序
- 没有推荐算法
- 内容被组织在它所属的实体（课程/组织/知识库）之下
- 搜索是意图驱动的——用户明确知道自己要找什么

### 5.4 克制与边界清晰

- 不提供即时聊天/群聊
- 不提供匿名发言
- 不提供点赞/转发/热度
- 不允许脱离上下文（具体实体）的自由发帖
- 讨论必须依附于具体对象

---

## 六、明确不做

- 即时聊天 / 群聊
- 全站信息流 / 热度榜
- 点赞 / 转发 / 关注机制
- 匿名发言 / 树洞 / 八卦
- 娱乐化积分/等级/徽章
- 接入教务系统直接选课（Phase 早期不做）
- App Push 通知（初期仅站内通知中心）
