# CampLus 产品定义

> 最后更新：2026-06-19
> 
> 本文档定义 CampLus **是什么、解决什么问题、由哪些核心系统组成**。
> 所有后续的路由、组件、数据模型、Phase 规划均从本文档派生。

---

## 一、CampLus 是什么

**一句话**：CampLus 是校园学术生活的操作系统——把学生与"课程、知识、选课、信息获取"相关的所有需求，从碎片化的 QQ/微信/Piazza/教务系统中抽出来，整合到一个结构化、可靠、不吵闹的平台里。

**核心哲学**：

1. **学生自治是战略选择**——平台在没有任何教师参与的情况下完整运转。课程信息由学生自主上传、集体编辑、集体审核、集体维护。教师是可选接入的"权威确认者"，不是系统运转的前提。
2. **利己是起点，共享是核心**——每个学生进来整理课程信息，最初是为了自己——这周有什么作业、哪天考试、截止时间是什么。这是起点，真实且无需说服。但当整理工具做得足够好用，个人整理的内容汇聚到一起，互相补充、互相纠错，信息的完整度和准确性就超出了任何单个人能做到的。**平台的核心价值不在个人整理，而在汇聚之后的样子。** 利己让用户进门，共享让平台成立。两者缺一不可。
3. **事实优先于情绪**——这是一个围绕"事实、时间与责任"构建的系统，不是表达情绪的社交平台。
4. **信息积累而非信息流**——内容被结构化存储、可引用、可追溯、可继承。不是"阅后即焚"的聊天记录。

## 二、解决的痛点

| # | 痛点 | 现状 | CampLus 的解法 |
|---|---|---|---|
| 1 | **课程信息碎片化** | 一门课的信息分散在 QQ 群公告、群聊天、Piazza、教务系统、教师个人主页 | 每门课一个结构化仪表盘，所有信息集中 |
| 2 | **行政通知被 QQ 淹没** | 辅导员/教务处/学院的通知和闲聊混在一起，重要信息被刷掉 | 学生自主汇总整理——看到通知的人把它录入平台，其他人受益 |
| 3 | **选课靠玄学** | 不知道老师的授课质量、给分分布、课程工作量、时间是否冲突 | 课程搜索引擎 + 档案页（成绩分布/评价聚合）+ Schedule Builder |
| 4 | **评课数据不流通** | 现有教学评估是行政导向的，数据不公开，对后来学生无参考价值 | 结构化评价系统，数据公开，支撑选课决策 |
| 5 | **社团信息孤岛化** | 每个社团一个 QQ 群+公众号，需要"打入内部"才能了解 | 组织公开主页 + 活动管理 + 公告发布 |
| 6 | **学生笔记锁在本地** | 好的笔记、总结、复习资料只存在于个人电脑或群文件里 | 未来由独立应用 **CampLus Note** 解决，与 CampLus 双向链接 |

## 三、五大核心系统

### 系统 A：课程仪表盘（Course Dashboard）

**解决痛点 #1**

每门课程拥有一个结构化主页。核心组成：

- **课程基本信息**：代码、名称、院系、学分、类别
- **学期列表**：历史学期进入只读归档，当前学期活跃
- **Timeline**：按教学周组织的节点（公告、作业、考试、资料、里程碑），是课程主页的信息核心
- **讨论**：围绕 Timeline 节点的上下文讨论（Q&A），不提供脱离节点的自由发帖
- **版本历史**：所有编辑记录 + Diff + 回滚

**编辑模式**：
- 学生自主上传和维护，标注信息来源（"课堂宣布于 3/15" / "学生整理"）
- 集体编辑 → 维护者审核合并（类似 GitHub PR 流程）
- 教师可标记"官方确认"，但不是必要环节
- 所有编辑永久可追溯

---

### 系统 B：个人首页（Personal Homepage / Dashboard）

**解决痛点 #6（个人学术全景缺失）**

登录后的默认页面。**仅展示与个人课程强相关的内容**：

- **我的课程卡片**：本学期在修课程，每门课显示下一个截止时间、本周要点
- **待跟进清单**：跨课程的作业和截止时间汇总
- **课程通知聚合**：被订阅课程的更新（新节点、修改、截止提醒）
- **学业进度总览**：已完成学分 / 总需学分 / 各模块完成情况

**明确不包含**：行政通知、全站信息流、推荐内容。

---

### 系统 C：选课 × 评课（Course Discovery & Evaluation）

**解决痛点 #3 + #4**

选课和评课是**一个闭环的两个方向**：评价产生数据 → 数据支撑选课 → 选课后的学期产生新评价。

**包含四个子模块**：

1. **课程搜索引擎**
   - 按课程名称/代码/院系/学分/类别/时间筛选
   - 模糊搜索 + 通配符支持
   - 搜索结果直达课程档案页

2. **课程档案页**（参考 Atlas）
   - 基本信息 + 历年开课记录
   - 历年成绩分布（各分数段比例）
   - 评分聚合（教学质量、课程内容、难度、工作量）
   - 教授评价聚合（清晰度、尊重度、备课充分度等维度）
   - 数据可视化呈现

3. **Schedule Builder**（参考 Atlas）
   - 将候选课程加入规划表
   - 可视化日历网格视图，生成所有可能的课表排列
   - 自动检测时间冲突
   - 锁定心仪时间段，屏蔽忙碌时段
   - 生成课表，导出/分享（暂不接入教务系统选课）

4. **教学评价系统**

   评价分为两层：**过程跟踪 + 最终评价**。

   **过程跟踪（学期中）**：
   - 修读期间可以随时记录对课程的印象、感受、笔记
   - 形式：私有的阶段性 check-in（打分草稿 + 文字笔记）
   - 可以随学期进展持续更新
   - 仅供自己回顾，不公开、不参与聚合

   **最终评价（学期结束后）**：
   - 触发条件：学期结束 + 用户标记为"已完整修读"
   - 评价结构：定量打分（匿名展示，系统可追溯）+ 定性论述（实名）
   - 约束：提交后不可修改、教师可回应不可删除
   - 数据用途：聚合为课程档案页的数据来源

   **两层关系**：过程跟踪是私人的、辅助性的——帮助学生积累素材，最终评价时可以参考自己之前的记录。最终评价才是正式数据。

---

### 系统 D：组织与社团（Organizations & Clubs）

**解决痛点 #5**

与课程系统同级的一级模块。

- **组织公开主页**：简介、成员、负责人、联系方式
- **活动管理**：每场活动是一个独立条目——标题、时间、地点、描述、状态（预告/进行中/已结束/取消）、可选报名/签到。活动之间没有阶段递进关系，**不复用课程 Timeline 结构**
- **组织公告**：社团的公开通知（招新、公告等），区别于活动（活动有时效性和地点，公告是纯信息发布）
- **成员可见列表**

**为什么不复用 Timeline**：课程 Timeline 的核心是"周→周→周"的递进结构，节点之间有先后依赖和累积关系。社团活动更接近"行政通知"——每场活动是独立的、离散的，没有"阶段1→阶段2→里程碑3"的叙事。

---

### 未来应用：CampLus Note（知识广场）

**解决痛点 #6（笔记锁在本地）**

知识广场与课程系统的用户行为和使用频率完全不同：
- 课程仪表盘是**被动消费 + 轻量编辑**，高频，紧迫
- 知识广场是**主动创作 + 深度协作**，低频，非紧迫

两者的技术形态也不同——知识广场需要 BlockNote 块级编辑器、GitHub 式 PR 协作流程、块级 diff/merge，技术复杂度远超课程系统。

因此**知识广场独立为 CampLus Note**，作为一个单独的应用和仓库：

- 独立开发、独立部署、独立域名
- 使用 BlockNote 驱动，Markdown + LaTeX 原生支持，GitHub 式协作
- 与 CampLus 主站通过**双向链接**关联：
  - 课程 Timeline 的 material 节点可以引用 CampLus Note 的笔记
  - CampLus Note 的笔记可以引用课程 Timeline 的节点
  - 课程主页展示"关联的社区笔记"
  - 个人首页可以 pin 自己的知识库

**CampLus Note 的优先级**：在 CampLus 核心课程系统验证可行之后再启动。它是锦上添花，不是地基。

---

### 系统 E：行政通知聚合（Administrative Notice Hub）

**解决痛点 #2**

**核心思路**：和学生自治一脉相承——不是让辅导员和教务处入驻平台发布通知，而是**学生看到通知后把它录入平台**。每个学生都是"信息的搬运工"。

**运作方式**：
- 任何学生都可以录入一条行政通知（来源、标题、内容、发布时间）
- 标注信息来源——"发布于计算机学院 2024 级 QQ 群，辅导员张老师"
- 其他学生可以补充、纠错、标注重要性
- 通知按来源（学院/教务处/学生处/辅导员）和类别（选课/考试/奖惩/后勤/活动）组织
- 不按时间流排列——按来源和类别结构化浏览
- 不在首页展示——独立模块，按需进入

**内容规则**：
- 只录入事实型通知（"期末考试安排在..."），不录入舆情、讨论
- 每条通知标注原始来源，可追溯
- 过期通知自动归档，仍可搜索

**为什么不自带官方发布功能**：一旦提供"校方发布端"，就回到了依赖学校配合的老路。学生自己搬运、自己整理——和在课程仪表盘里维护 Timeline 是同一个逻辑。

## 四、系统之间的关系

```
个人首页 ──→ 聚合来自"课程仪表盘"的更新
   │
   ├──→ 点击课程卡片进入 → 课程仪表盘
   │                          ├─ Timeline 节点 ──→ 链接到 → CampLus Note（外部应用）
   │                          ├─ Material 节点 ──→ 引用自 → CampLus Note
   │                          └─ 讨论（围绕节点）
   │
   ├──→ 选课×评课
   │        ├─ 课程搜索 → 课程档案页（成绩+评价数据）
   │        ├─ Schedule Builder（可视化排课）
   │        └─ 教学评价 → 数据流入课程档案页
   │
   ├──→ 组织与社团
   │        └─ 活动与公告（独立数据模型）
   │
   └──→ 行政通知聚合
            └─ 学生录入 + 按来源/类别组织 + 归档

CampLus Note（外部独立应用）
   └─ 笔记 ──→ 双向链接 → CampLus 课程 Timeline 节点
```

**关键交互**：
- 课程 ↔ CampLus Note 是**双向链接**关系，跨应用引用。两者互不隶属
- 选课×评课的数据来源于教务数据导入 + 评价系统，产出到课程档案页
- Timeline 是课程系统的核心数据结构，但不被组织/社团模块复用——不同实体有不同的时间组织方式
- 行政通知聚合采用独立的数据模型

## 四之附、数据来源策略

> 核心原则：**最小化学校依赖**。CampLus 不"对接"学校系统，而是学生用工具把自己合法获取到的数据带进来。

### 社区自产数据（零依赖）

以下数据由平台自身产生，不依赖任何外部系统：

| 数据 | 来源 |
|---|---|
| 课程 Timeline 内容 | 学生编辑维护 |
| 讨论与评论 | 学生讨论 |
| 知识广场笔记 | 学生创建 |
| 教学评价 | 学生提交 |
| 组织与社团信息 | 社团成员维护 |
| 行政通知 | 学生从 QQ 群等渠道搬运录入 |

### 教务数据（需外部获取）

选课×评课系统需要的数据无法由社区凭空生产。获取策略：

| 数据 | 获取方式 | 说明 |
|---|---|---|
| 课程目录（代码、名称、学分、院系、开课时间、地点、教师） | 课表导入（参考 WakeUp 课表）+ 爬虫补充 | 核心策略：**让学生用最短路径把自己的课表导进来**。参考 WakeUp 课表的做法——从教务系统截图识别、或通过学生凭证拉取课程数据。课表导入是冷启动的关键第一步：学生为了自己方便导入课表 → 平台获得课程数据 → Schedule Builder 有数据可用。导入体验必须做到"打开 → 授权/截图 → 完成"，三步以内 |
| 历年成绩分布 | 爬虫（如有公开数据） | 部分大学教务系统提供成绩分布查询（类似 UMich），如有则可爬取 |
| 课程先修关系 | 爬虫或手动录入 | 培养方案中的先修课程信息 |

**爬虫的原则**：
- 使用学生自己的凭证访问，不涉及任何未授权操作
- 爬取的是学生合法有权查看的数据
- 爬虫作为独立工具，不属于平台核心代码（降低风险）
- 数据导入平台后，后续维护由社区承担

### 明确不依赖学校的

- 不依赖学校 IT 部门提供 API
- 不依赖校方签署合作协议
- 不依赖教师入驻或行政人员发布内容
- 平台运转的唯一外部数据需求是**课程目录的结构化导入**——而这是学生用自己的账号就能获取的数据

## 五、核心设计原则

### 5.1 学生自治

> **平台在没有任何教师参与的情况下完整运转。**

- 课程信息由学生上传、编辑、审核、维护
- 教师是可选接入的"权威确认者"，可以标注官方信息、修正事实错误，但：
  - 不能删除学生编辑的内容
  - 不能删除评价
  - 不能禁止讨论
- 这样做的原因不只是理想——是**战略选择**：一旦系统运转依赖学校行政配合，就会被卡住

### 5.2 责任可追溯

- 所有编辑记录编辑者、时间、修改说明
- 历史版本不可删除
- 讨论实名
- 任何获得更大权限的角色，必须同时承担更明确、可追溯的责任

### 5.3 信息结构化而非信息流

- 没有全站信息流
- 没有热度排序
- 没有推荐算法
- 内容被组织在它所属的实体（课程/组织/知识库）之下
- 搜索是意图驱动的——用户明确知道自己要找什么

### 5.4 克制与边界清晰

- 不提供即时聊天/群聊
- 不提供匿名发言
- 不提供点赞/转发/热度
- 不允许脱离上下文（具体实体）的自由发帖
- 讨论必须依附于具体对象

---

## 六、明确不做

- 即时聊天 / 群聊
- 全站信息流 / 热度榜
- 点赞 / 转发 / 关注机制
- 匿名发言 / 树洞 / 八卦
- 娱乐化积分/等级/徽章
- 接入教务系统直接选课（Phase 早期不做）
- App Push 通知（初期仅站内通知中心）
